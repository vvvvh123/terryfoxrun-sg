Executive Committee Proposal

Terry Fox Run Singapore Website Proposal

Purpose: Initial approval of website direction, feature scope, and implementation priorities

We recommend continuing with the product direction developed so far: a Terry Fox Run Singapore website that brings event registration, optional T-shirt purchase, donations, corporate shirt orders, participant confirmations, pickup QR codes, announcements, and admin operations under our own control.

Goals and Success Criteria

The website should help the committee keep more fundraising value, make participant registration clearer, support volunteer operations, and make future yearly runs easier to launch.

  • Participants can complete registration on mobile without confusion.
  • One payer can register multiple participants in a single checkout.
  • Shirt purchases and donations are clearly separated from run participation.
  • T-shirt pickup can be handled by QR code or manual code.
  • Admins can configure each event without changing code.

Proposed Page Map

Public Pages

  • Home: Event overview, mission, Register CTA, Donate CTA, T-shirt preview, route highlights, sponsors, FAQ entry point.
  • Event Details: 5K and 10K information, schedule, venue, registration close date, pickup dates, FAQ.
  • Register: Individual registration with support for adding multiple participants.
  • Corporate Orders: Company shirt-order form for bulk shirt quantities, with offline payment follow-up.
  • Confirmation: Registration summary, payment status, receipt information, pickup QR/code, next steps.
  • Past Runs: Archive of previous runs, highlights, photos, and links.

Participant and Admin Areas

  • Dashboard: My registrations, QR/pickup code, T-shirt order details, donation details, receipts, announcements.
  • Admin: Event setup, field configuration, inventory, registrations, corporate orders, pickup, announcements, exports, and overview metrics.

Registration Model

The individual registration flow should support one payer registering multiple participants. Per participant, configurable fields can include name, email, phone, emergency contact, date of birth, gender, address, residency or identity information, medical notes, and category.

The admin should be able to set each field as required, optional, or hidden per event. Run participation itself should not require a T-shirt purchase.

For v1 payments, individual registrations should support PayNow and bank transfer. Both methods are manually verified by admins before the registration is fully confirmed.

T-Shirt Purchase and Pickup Model

T-shirt purchase should be optional. The default T-shirt price is SGD 35 per shirt, regardless of adult or kids size.

Default sizes are Adult XS to XXXL and Kids XS to L. Admins should be able to configure which sizes are available for each event and set inventory quantities.

  • After successful payment, each participant or order receives a unique QR code and short pickup code.
  • The QR/code appears in the confirmation email and participant dashboard.
  • Volunteers can scan the QR code or enter the code manually.
  • The system marks the shirt as collected and prevents duplicate collection.
  • T-shirt size changes are not allowed after purchase.
  • A designated collector can use the participant's QR confirmation slip to collect on their behalf.

Donation Model

Donation should be encouraged but optional. Suggested presets are SGD 20, SGD 50, and SGD 100, with a custom amount option.

Corporate Order Model

Corporate orders should be shirt-only for v1. They should not require individual runner details.

  • Collect company name, company address, UEN, contact name, contact email, contact phone, and shirt quantities.
  • Skip online payment and follow up separately with invoice or bank transfer instructions.
  • Allow admins to view, update status, and export corporate orders.

Admin Capabilities

Required roles are Admin, Volunteer, and Participant.

  • Admins can manage event setup, field requirements, categories, inventory, announcements, corporate orders, pickup, reports, and exports.
  • Volunteers can use the pickup scanning and manual code entry tool.
  • Participants can access their own dashboard and registrations.

Yearly Reuse Model

The website should be designed around event configuration rather than one-off hardcoded pages. Each yearly event can have its own dates, locations, categories, field rules, donation presets, shirt price, inventory, homepage copy, and visual assets.

The organizing team should be able to clone the previous year's event, update details, set inventory, and open registration. Past events should remain available on a Past Runs page.

Reference Facts From Existing Public Material

  • The 2025 Singapore listing describes the run as taking place on 30 Nov 2025 at Angsana Green ECP at 7am, with 5km and 10km options and no entry fee.
  • The existing FAQ states that there are no race packs, bibs, timing chips, winners, or prizes.
  • The existing FAQ states that no T-shirt size exchanges are allowed.
  • The existing FAQ says a designated person can use the QR confirmation slip to collect a shirt.

Sources: JustRunLah 2025 listing, Terry Fox Run Singapore FAQ, and existing registration reference.

Decisions Requested From the Committee

DecisionApprove the overall visual direction: red, white, black, dark blue, warm, modern, and event-focused.

DecisionApprove MVP scope: public site, registration, optional shirts, optional donations, corporate orders, dashboard, admin, pickup, announcements, and exports.

DecisionApprove PayNow and bank transfer as v1 payment methods, with manual admin verification.

DecisionApprove corporate orders as shirt-only with offline payment follow-up.

DecisionApprove pickup model: QR code plus manual code, no size exchanges, designated collector allowed.

Recommended Implementation Phases

  1. Phase 1: Approval, copy, assets, technical cleanup, final event facts, and technical stack confirmation.
  2. Phase 2: Public site, registration, PayNow/bank-transfer checkout instructions, payment proof submission, corporate form, pending emails, and dashboard basics.
  3. Phase 3: Admin portal, manual payment verification, inventory, pickup scanning, corporate packages/orders, mass email, exports, and event-day testing.
  4. Later: sponsor-a-participant, lucky draw, Stripe or other automated payment integration, SMS, advanced offline pickup sync, and analytics.

Technical Stack Summary

The full technical stack is documented in technical-stack-and-design-plan.md.

  • Frontend: Next.js App Router, TypeScript, Material UI, React Hook Form, Zod, TanStack Query, Recharts, QR display/scanning libraries.
  • Backend: Java 21, Spring Boot 3.x, REST APIs, Spring Security with Auth0 JWT validation, Spring Data JPA, PostgreSQL, Flyway, Actuator.
  • Authentication: Auth0 Universal Login with participant, admin, and volunteer roles.
  • Payments: manual PayNow and bank-transfer verification for launch.
  • Storage: managed object storage for payment proof uploads, slideshow images, indemnity files, and exports.
  • Hosting: Vercel frontend, Fly.io Singapore backend, and Supabase Singapore database/storage.
  • Email: Postmark or SendGrid with SPF, DKIM, and DMARC configured before launch.
