Technical Plan

Terry Fox Run Singapore Technical Stack and Design Plan

Use a managed, conservative stack: Next.js + Material UI frontend, Spring Boot Java backend, PostgreSQL, Auth0, manual PayNow/bank-transfer reconciliation, admin-managed media, and hosted email. The architecture prioritizes mobile UX, yearly reuse, auditability, and low operational burden.

Recommended Stack

Frontend

  • Next.js App Router with TypeScript.
  • Material UI with Emotion styling.
  • React Hook Form + Zod for forms and validation.
  • TanStack Query for backend REST API data fetching.
  • Recharts for fundraising and inventory trend charts.
  • QR code display and browser camera QR scanning for pickup workflows.
  • Use MUI only; remove accidental Tailwind setup.

Backend

  • Java 21 and Spring Boot 3.x.
  • REST JSON APIs.
  • Spring Security OAuth2 Resource Server validating Auth0 JWTs.
  • Spring Data JPA / Hibernate with PostgreSQL.
  • Flyway migrations and Jakarta Bean Validation.
  • Spring scheduled jobs for payment reminders, email retries, and admin summaries.
  • Spring Actuator and structured logs for observability.

Database, Storage, Auth, Hosting, Email

  • PostgreSQL, recommended on Supabase Singapore region.
  • Supabase Storage or equivalent object storage for proof uploads, slideshow images, indemnity files, and exports.
  • Auth0 Universal Login with participant, admin, and volunteer roles.
  • Frontend on Vercel; backend on Fly.io Singapore; database/storage on Supabase Singapore.
  • Postmark or SendGrid for email, with SPF, DKIM, and DMARC configured before launch.

Key Technical Decisions

Payments

  • Initial launch uses manual PayNow and bank-transfer verification.
  • System generates a unique order/payment reference.
  • User submits transaction/payment ID and optional proof upload after clicking "I've paid."
  • Admin verifies payment, records final transaction ID, and marks order paid.
  • Fundraising tracker and pickup QR/code depend on admin-confirmed payments.

Configurable Events

  • Admin configures current event, event detail page, categories, form fields, indemnity text, T-shirt sizes, inventory, payment instructions, and homepage slideshow.
  • Homepage shows current-event overview only; event detail page carries full details.
  • Slideshow supports uploaded images plus blurbs/captions.
  • Past event and transaction history remain available across years.

Registration, Corporate, Inventory, Admin

  • Payer account is required; payer fields are admin-configurable.
  • Additional participant details are optional by default; category is per participant.
  • Corporate packages, shirt allocations, payment proof, and status history are stored and exportable.
  • Every inventory change creates an inventory movement record for daily trend charts.
  • Admin portal covers payment verification, slideshow, form fields, categories, packages, inventory, pickup, exports, mass email, and dashboards.

Data and Interfaces

  • EventSlideshowImage: image URL, blurb, display order, active flag.
  • EventFormFieldConfig: field key, label, required/optional/hidden, applies-to scope.
  • PaymentAttempt: method, generated reference, user transaction ID, proof file URL, verification status.
  • InventoryMovement: event, size, type, quantity delta, reason, linked order, timestamp.
  • CorporatePackage: package name, price, shirt allocation rules.
  • EmailCampaign: audience, subject, body, sent status, created by.

Test Plan

  • Unit tests for pricing, donations, shirt quantities, payment transitions, and inventory movements.
  • PostgreSQL/Testcontainers integration tests for registration, corporate orders, payment verification, and pickup.
  • Playwright end-to-end tests for mobile registration, proof submission, admin verification, My Events, pickup, and slideshow updates.
  • Email provider sandbox tests, role/permission tests, and accessibility checks.
